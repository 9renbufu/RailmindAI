## 一、分层防护架构

```
┌─────────────────────────────────────────────────────────────┐
│                        用户请求                               │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
                    ┌──────────────────────┐
                    │    CDN / 静态资源     │     ← L1: 边缘缓存
                    └──────────┬───────────┘
                               ▼
                    ┌──────────────────────┐
                    │   Nginx 反向代理      │     ← L2: 负载均衡
                    │   限流 / IP黑名单     │
                    └──────────┬───────────┘
                               ▼
                    ┌──────────────────────┐
                    │    API Gateway       │     ← L3: 网关限流
                    │    全局/接口/用户限流  │        Sentinel / 自研
                    └──────────┬───────────┘
                               ▼
                    ┌──────────────────────┐
                    │    服务层限流          │     ← L4: 服务级防护
                    │    Redis计数器        │
                    └──────────┬───────────┘
                               ▼
                    ┌──────────────────────┐
                    │    业务层防护          │     ← L5: 业务级
                    │    库存预扣 / 队列化   │
                    └──────────┬───────────┘
                               ▼
                    ┌──────────────────────┐
                    │    数据层              │     ← L6: 数据层
                    │    Redis + MySQL      │
                    └──────────────────────┘
```

## 二、核心场景高并发方案

### 2.1 余票查询（QPS 50,000+）

```
┌─────────────┐
│  本地缓存    │  Caffeine Cache, TTL 5s
│  (L1)       │  热门线路命中率 > 80%
└──────┬──────┘
       │ miss
       ▼
┌─────────────┐
│  Redis缓存   │  TTL 30s
│  (L2)       │  余票快照 Hash
└──────┬──────┘
       │ miss
       ▼
┌─────────────┐
│  MySQL       │  读写分离
│  (L3)       │  从库查询
└─────────────┘
```

- 本地缓存使用 Caffeine，热门线路（京沪、京广等）预热
- Redis 使用 Pipeline 批量查询
- 查询结果使用 Bloom Filter 预判：无票直接返回，不查库

### 2.2 下单/秒杀（QPS 5,000+，不超卖）

```
Redis + Lua 原子扣减 + 异步下单

┌──────────┐    ┌─────────────┐    ┌──────────┐    ┌──────────┐
│  用户    │───>│ Redis+Lua   │───>│  Kafka   │───>│  DB      │
│  请求    │    │ 原子扣库存   │    │  异步    │    │  落库    │
└──────────┘    └─────────────┘    └──────────┘    └──────────┘

Lua脚本逻辑：
1. 检查库存是否充足
2. 检查用户是否重复下单
3. 扣减库存
4. 记录用户下单标记
5. 返回结果

只有 Redis 扣减成功的请求才会发往 Kafka，极大降低 DB 压力。
```

### 2.3 座位锁定

```
方案：Redis 分布式锁 + 数据库乐观锁

锁定流程：
1. Redis SETNX seat:lock:{trainId}:{date}:{seatNo} = {orderNo} EX 900
2. 成功 → 写 t_seat_lock 表（status=1）
3. 失败 → 返回"座位已被占用"

释放流程：
1. 删除 Redis Key
2. 更新 t_seat_lock (status=0)
3. 发送 ticket-released 事件

防死锁：
- Redis Key TTL 15分钟兜底
- 定时任务扫描 t_seat_lock 中 expire_time 已过期的记录
```

### 2.4 候补购票

```
方案：Redis ZSet + 定时扫描

入队：ZADD waitlist:{trainId}:{date}:{from}:{to}:{seatType} {priority} {waitlistId}
出队：ZPOPMAX（优先级最高）
扫描：每30秒扫描有库存的线路，批量出队 → 创建订单

优先级计算：
- 用户等级（VIP加分）
- 候补时间（先到先得）
- 候补天数（多天候补加分）
```

## 三、限流策略

| 层级 | 策略 | 参数 |
|------|------|------|
| Gateway | 令牌桶 | 全局 100,000 QPS |
| Gateway | 滑动窗口 | 单接口 10,000 QPS |
| 服务层 | 滑动窗口 | 单用户 100 次/分钟 |
| 下单 | 计数器 | 单用户单车次 1次/秒 |
| 查询 | 令牌桶 | 单用户 200次/分钟 |

## 四、熔断降级

| 场景 | 熔断条件 | 降级策略 |
|------|---------|---------|
| 余票查询 | 错误率 > 50% (10s窗口) | 返回本地缓存 / 提示稍后重试 |
| 下单服务 | 错误率 > 30% | 进入排队页，稍后重试 |
| 支付服务 | 超时 > 3s | 异步支付，轮询结果 |
| AI 服务 | 超时 > 5s | 关闭 AI 推荐，返回基础结果 |
| 通知服务 | 错误率 > 80% | 降级为异步批量推送 |

## 五、数据库优化

| 策略 | 说明 |
|------|------|
| 读写分离 | 查询走从库，写操作走主库 |
| 连接池 | HikariCP，最大连接数 50 |
| 慢查询 | 超过 200ms 自动记录 |
| 索引优化 | 核心查询全部覆盖索引 |
| 批量操作 | 库存更新使用批量扣减 |
